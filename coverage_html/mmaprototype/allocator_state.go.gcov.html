     430              : // The logic in sortTargetCandidateSetAndPick related to load is a heuristic
     431              : // motivated by the following observations:
     432              : //
     433              : // 1. The high level objective is mostly to move towards the mean along all
     434              : // resource dimensions. For now, we are fuzzy on whether this should be the
     435              : // usage mean or the utilization mean, since historically the allocated has
     436              : // considered the former only, while mmaprototype in loadSummaryForDimension considers
     437              : // both.
     438              : //
     439              : // 2. We want to minimize range movement (and to a lesser extent lease
     440              : // movement) when moving towards the mean, so it is preferable to first move
     441              : // to stores that are not overloaded along any dimension, since they are
     442              : // guaranteed to not have to shed the moved load. Hence sorting based on the
     443              : // overall store loadSummary first for the targets.
     444              : //
     445              : // 3. It is possible (as also demonstrated by asim tests) that with multiple
     446              : // metrics, we can have scenarios where every store is loadNoChange or
     447              : // overloaded along some dimension while underloaded along another dimension
     448              : // (the latter is necessarily true if overload and underload are based on the
     449              : // mean). We have to be able to rebalance out of such local optima. One way to
     450              : // do so would be introduce the ability to decide changes to multiple ranges
     451              : // at once, however this adds code complexity. Instead, we can ignore the fact
     452              : // that the target is overall overloaded, and pick a candidate that is most
     453              : // underloaded in the dimension that the source wants to shed. This should
     454              : // allow the target to accept the range and later shed some other range. There
     455              : // is the possibility that will result in more range movement until we reach
     456              : // the goal state, but so be it.
     457              : 
     458              : // In the set of candidates it is possible that some are overloaded, or have
     459              : // too many pending changes. It divides them into sets with equal diversity
     460              : // score and sorts such that the set with higher diversity score is considered
     461              : // before one with lower diversity score. Then it finds the best diversity set
     462              : // which has some candidates that are not overloaded wrt disk capacity. Within
     463              : // this set it will exclude candidates that are overloaded or have too many
     464              : // pending changes, and then pick randomly among the least loaded ones.
     465              : //
     466              : // Our enum for loadSummary is quite coarse. This has pros and cons: the pros
     467              : // are (a) the coarseness increases the probability that we do eventually find
     468              : // something that can handle the multidimensional load of this range, (b)
     469              : // random picking in a large set results in different stores with allocators
     470              : // making different decisions, which reduces thrashing. The con is that we may
     471              : // not select the candidate that is the very best. Normally, we only consider
     472              : // candidates in the best equivalence class defined by the loadSummary
     473              : // aggregated across all dimensions, which is already coarse as mentioned
     474              : // above. However, when ignoreHigherThanLoadThreshold is set and an
     475              : // overloadedDim is provided, we extend beyond the first equivalence class, to
     476              : // consider all candidates that are underloaded in the overloadedDim.

     481              : // considered.
